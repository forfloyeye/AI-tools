## 一、项目背景

一款主打"**即用即走、纯内存流转、不存原图、按点数计费**"的电商卖家图片处理 Web 应用。
当前阶段为**前端 Mock 阶段**，所有 AI 调用和积分扣费均为前端模拟，后续将接入真实后端与第三方 API。

## 四、路由设计

| 路由 | 组件 | 说明 |
|---|---|---|
| `/` | `Dashboard` | 工作台大厅，展示所有工具卡片 |
| `/tools/remove-bg` | `RemoveBg` | 智能抠图工具页 |
| `/tools/ai-scene` | `AiScene` | AI 商品图工具页 |

> ⚠️ **无独立 `/editor` 路由**。编辑功能（尺寸预设、背景切换）以状态切换方式内嵌于各工具页面 `success` 状态，不跳转新页面。

页面切换动画：`opacity + scale(0.98)` 淡入淡出，duration 200ms。

## 五、全局状态设计（AppContext）

```typescript
interface AppContextType {
  user: { email: string; avatar: string } | null;
  points: number;                        // 单一积分字段（登录赋 300）
  isAuthModalOpen: boolean;
  authModalClosable: boolean;            // false = 被动拦截，不可关闭
  toast: { message: string; type: 'success' | 'error' } | null;

  login: (email: string) => void;        // 设置用户 + 赋 300 点
  logout: () => void;                    // 清空用户和积分
  deductPoints: (amount: number) => boolean; // 失败时自动 showToast 错误
  openAuthModal: (closable?: boolean) => void;
  closeAuthModal: (force?: boolean) => void;
  showToast: (message: string, type?) => void; // 3秒自动消失
}
```

**积分规则（当前 Mock 实现）**：
- 登录时 `setPoints(300)`
- 扣费：`setPoints(prev => prev - amount)`（前端状态减法，非原子操作）
- 不足时：Toast 提示"点数不足，请充值"，返回 `false`，终止操作

## 六、核心页面逻辑

### 6.1 智能抠图（RemoveBg）

```
状态: 'idle' | 'processing' | 'success'
```

| 步骤 | 逻辑 |
|---|---|
| 选择文件 | `accept="image/*"`，无严格格式限制，UI 提示 10MB |
| 触发处理 | 立即 `setStatus('processing')`，展示扫描线动画 |
| Mock 2000ms | `setTimeout` 后调用 `deductPoints(10)` |
| 成功 | `setStatus('success')`，右侧控制面板解锁 |
| 失败（点数不足） | Toast 提示 + `navigate('/')`，跳回工作台大厅 |
| 重置 | 清空 `image`、`aspectRatio`、`bgColor`，回到 `idle` |
| 下载 | `URL.createObjectURL` + `<a>` 触发下载 |

### 6.2 AI 商品图（AiScene）

```
状态: 'idle' | 'processing' | 'success'
```

| 步骤 | 逻辑 |
|---|---|
| 未选场景时拖/点上传 | 触发 `shake` 动画，显示警告 |
| 选中场景+上传图片 | `setStatus('processing')`，2500ms 后扣 30 点 |
| 失败（点数不足） | Toast 提示 + `navigate('/')`，跳回工作台大厅 |
| `success` 状态下切换场景 | 自动重新触发 `processing`（再扣 30 点） |
| 重置 | 清空 `image`，回到 `idle`，**保留已选场景** |

## 八、绝对红线（后端接入时必须遵守）

以下规范在当前 Mock 实现中未涉及，**后端接入时必须严格执行**：

### 8.1 零磁盘持久化
- 绝对禁止将用户上传原图、生成中间图、API 返回结果图持久化到服务器磁盘或数据库
- 必须纯内存或临时 Blob 数据流转

### 8.2 防 API 超时
- 调用耗时第三方 API（AI 生图，预计 15-20 秒）时，前端必须有全屏 Loading 遮罩 + 防抖，防止二次提交
- 后端接口必须设置足够的超时时间（≥60s）

### 8.3 并发安全与原子扣费
- 所有扣费动作必须使用数据库原子递减操作（如 SQLite 事务），**严禁**内存加减后再 Update
- 如第三方 API 调用失败，必须在 `catch` 块内原子回滚积分，并向前端返回明确错误提示

## 九、后端规划（待实现）

### 9.1 数据库模型（SQLite / 可迁移至 SQL Server）

```sql
-- User 表
CREATE TABLE User (
  id           TEXT PRIMARY KEY DEFAULT (lower(hex(randomblob(16)))),
  email        TEXT UNIQUE NOT NULL,
  passwordHash TEXT NOT NULL,          -- Bcrypt 哈希
  freeCredits  INTEGER DEFAULT 300,    -- 每日免费点数
  paidCredits  INTEGER DEFAULT 0,      -- 充值付费点数
  lastLoginDate DATETIME DEFAULT CURRENT_TIMESTAMP,
  createdAt    DATETIME DEFAULT CURRENT_TIMESTAMP
);

-- RedeemCode 表（卡密充值）
CREATE TABLE RedeemCode (
  id          TEXT PRIMARY KEY,
  code        TEXT UNIQUE NOT NULL,
  creditValue INTEGER NOT NULL,
  isUsed      INTEGER DEFAULT 0,       -- Boolean: 0/1
  usedAt      DATETIME,
  usedById    TEXT REFERENCES User(id),
  createdAt   DATETIME DEFAULT CURRENT_TIMESTAMP
);
```

### 9.2 核心业务流

#### 智能抠图流（消耗 10 点）
1. 校验 `freeCredits + paidCredits >= 10`
2. **事务内原子扣费**（优先扣 freeCredits，不足扣 paidCredits）
3. 调用第三方抠图 API
4. **补偿机制**：API 失败 → `catch` 内原子退还 10 点 → 返回 500 错误
5. 成功 → 返回透明图数据流给前端

#### AI 商品图流（消耗 30 点）
1. 校验总点数 >= 30 并原子扣费
2. **串联 API A**：调用抠图 API 获取透明主体图
3. **串联 API B**：透明图 + 场景 Prompt → 发给第三方生图 API
4. **补偿机制**：A 或 B 任何一步失败 → `catch` 内原子退还 30 点 → 抛出异常
5. 成功 → 返回合成图给前端

#### 卡密充值流（事务）
1. 校验 `RedeemCode` 中 `code` 存在且 `isUsed = 0`
2. **事务**：`paidCredits += creditValue` + 标记卡密 `isUsed = 1` + 记录 `usedAt`、`usedById`
3. 任何步骤失败则回滚

### 9.3 鉴权（待实现）
- 方案：Credentials（邮箱+密码），Bcrypt 哈希验证
- 会话：JWT 或 Session Cookie
- 每日积分重置：登录或访问 Dashboard 时，后端校验 `lastLoginDate`，如不是今天则将 `freeCredits` 重置为 300
